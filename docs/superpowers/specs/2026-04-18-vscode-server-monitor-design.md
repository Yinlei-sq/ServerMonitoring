# VS Code 服务器监控插件设计

## 1. 目标

开发一个 VS Code 监控插件，满足以下核心体验：

- 未连接远程服务器时，默认监控本机资源
- 连接 VS Code Remote SSH 后，默认监控当前远程 Linux 服务器资源
- 在远程窗口中，用户仍可手动切换回本机监控
- 提供活动栏仪表盘视图和资源管理器树视图
- 第一版支持 CPU、内存、磁盘、网络、系统负载、运行时长、GPU、进程列表与操作按钮

## 2. 非目标

第一版不包含以下能力：

- 多服务器资产管理
- 历史监控数据存储与趋势图
- 告警通知与阈值规则
- Docker、Kubernetes、容器级监控
- 独立常驻 agent
- 默认内置高风险系统命令

## 3. 总体方案

采用 VS Code 双宿主扩展架构。

- UI 运行在本地 VS Code 侧
- 本机采集逻辑运行在本地扩展宿主
- 远程采集逻辑运行在 Remote SSH 远程扩展宿主
- 统一由控制器管理“当前监控目标”为本机或远程

该方案优先复用 VS Code 原生 Remote SSH 运行环境，不重复实现 SSH 会话和凭据管理，也不要求服务器部署额外 agent。

## 4. 用户体验

### 4.1 监控目标切换

- 本地窗口：默认目标为本机
- Remote SSH 窗口：默认目标为远程服务器
- 远程窗口中提供“本机 / 当前远程”切换入口
- 用户手动切换后，当前窗口记住选择，直到上下文变化或用户再次切换

### 4.2 视图入口

第一版同时提供两种入口：

- 活动栏独立视图容器：主监控仪表盘
- Explorer 树视图：展示关键摘要和快捷操作

### 4.3 主界面布局

活动栏 Webview 仪表盘按以下结构组织：

- 顶部栏：`性能 / 进程` Tab、最后更新时间、运行状态、设置入口
- 目标栏：当前目标、本机与远程切换
- 概览区：CPU、RAM、磁盘、网络
- GPU 区：多卡概览与单卡详情
- 进程区：搜索、排序、详情、结束进程
- 操作区：清理缓存、重新探测环境、自定义命令

UI 风格参考用户提供截图，但第一版以结构一致、信息清晰和交互可用为优先，不追求完全像素复刻。

## 5. 功能范围

### 5.1 必做监控项

- CPU 总利用率
- CPU 核心数
- 1 / 5 / 15 分钟系统负载
- 内存总量、已用、可用、使用率
- 磁盘挂载点、总量、已用、可用、使用率
- 网络上行 / 下行速率
- 系统运行时长
- 进程列表

### 5.2 GPU

如果系统可用 `nvidia-smi`，展示：

- GPU 型号
- 利用率
- 显存使用量和占比
- 温度
- 功耗

如系统无 GPU 或缺少命令，GPU 卡片单独降级为不支持状态，不影响其他模块。

### 5.3 进程视图

进程视图至少支持：

- 按 CPU 占用排序
- 按内存占用排序
- 按名称搜索
- 查看 PID、用户、CPU、内存、启动时间、命令行
- 结束进程

## 6. 操作边界

第一版允许轻量操作，但必须显式受限和可解释。

### 6.1 结束进程

- 本机和远程均支持
- 必须进行确认
- 对失败结果展示明确错误信息

### 6.2 清理缓存

- 不做“自动智能清理”
- 仅支持用户在设置中配置安全命令模板
- Linux 可配置如缓存回收类命令
- Windows 和 macOS 默认不提供高风险系统操作

### 6.3 重新探测环境

该动作定义为重新检测运行时环境，而不是修改系统环境变量。行为包括：

- 重新检测监控能力
- 重新检查 GPU 命令可用性
- 重新加载自定义命令能力状态

### 6.4 自定义命令

- 仅允许设置中的白名单命令
- 按按钮执行前展示目标环境
- 展示标准输出 / 标准错误摘要
- 不开放任意命令输入框作为默认功能

## 7. 兼容性边界

### 7.1 本机

支持：

- Windows
- macOS
- Linux

### 7.2 远程

仅支持：

- Linux

默认依赖以下常见能力：

- `sh`
- `ps`
- `df`
- `free` 或等价内存信息来源
- `uptime`
- `/proc`
- `nvidia-smi`（可选）

如部分命令不存在，插件以模块级降级方式处理。

## 8. 数据模型

本机和远程统一输出 `MonitorSnapshot`：

- `target`: `local` 或 `remote`
- `host`: 主机名、平台、架构、运行时长、时间戳
- `cpu`: 利用率、核心数、负载
- `memory`: 总量、已用、可用、使用率
- `disks`: 挂载点列表
- `network`: 上行 / 下行速率
- `gpus`: GPU 列表
- `processes`: 进程列表
- `capabilities`: 当前环境支持的操作能力
- `status`: 各模块状态

每个模块状态允许以下值：

- `ok`
- `unsupported`
- `permissionDenied`
- `commandFailed`

## 9. 刷新策略

为降低本机与远程负载，采用分层刷新：

- 总览卡片：每 2 秒刷新
- 进程列表：每 5 秒刷新
- 磁盘信息：每 15 秒刷新
- 用户执行操作后：立即触发一次强制刷新

控制器对最近一次快照做缓存，避免 Webview 和树视图重复触发采集。

## 10. 架构拆分

建议代码结构如下：

- `src/extension.ts`
- `src/core/monitorController.ts`
- `src/core/targetResolver.ts`
- `src/shared/types.ts`
- `src/shared/protocol.ts`
- `src/collectors/local/`
- `src/collectors/local/windows/`
- `src/collectors/local/macos/`
- `src/collectors/local/linux/`
- `src/collectors/remote/linux/`
- `src/collectors/remote/linux/commands/`
- `src/collectors/remote/linux/parsers/`
- `src/actions/`
- `src/webview/`
- `src/views/`
- `src/config/`

### 10.1 Extension Core

负责：

- 扩展激活
- 注册视图、命令、设置
- 连接 Webview 与树视图
- 管理控制器生命周期

### 10.2 MonitorController

负责：

- 判断当前目标
- 刷新调度
- 快照缓存
- 错误聚合
- 将统一数据推送给 UI

### 10.3 Collectors

分为：

- `LocalCollector`
- `RemoteCollector`

两者输出相同结构，不让 UI 关心数据来源。

### 10.4 Remote Linux Collector

远程采集分为两层：

- `commands`: 构造命令
- `parsers`: 解析标准输出

避免在控制器里混杂命令拼接和数据解析逻辑。

### 10.5 Actions

负责执行用户操作：

- 结束进程
- 重新探测环境
- 执行自定义命令
- 调用缓存清理模板命令

## 11. 本机与远程采集策略

### 11.1 本机

按平台实现采集：

- Windows：优先 PowerShell 与系统命令
- macOS：基于 `top`、`ps`、`df`
- Linux：基于 `/proc`、`ps`、`df`

### 11.2 远程 Linux

远程采集器运行在 VS Code Remote SSH 的远程扩展宿主中，直接在远程环境执行采集命令，不自行维护 SSH 链接。

### 11.3 本机数据在远程窗口中的获取

远程窗口下仍需支持“切回本机”。因此需要一条本地数据通道。实现上以本地扩展宿主采集本机数据，再与远程宿主数据在统一控制器协议中汇合。

实现重点：

- 保证本机采集和远程采集均可在当前窗口拿到
- 切换目标时只切换数据源，不重建 UI
- 如果远程断开，自动回退到本机

## 12. 错误处理

采用“局部失败不影响整体”的策略：

- GPU 失败仅影响 GPU 卡片
- 磁盘失败仅影响磁盘卡片
- 进程命令超时仅影响进程列表
- 自定义命令失败仅反馈给操作面板

需要展示的错误类型：

- 命令不存在
- 权限不足
- 命令超时
- 远程会话断开
- 解析失败

## 13. 配置项

第一版至少提供以下设置：

## 16. 关键风险

### 16.1 VS Code 双宿主通信复杂度

风险：本机数据与远程数据在同一窗口下的协调实现不当，会导致切换体验混乱。

缓解：

- 先定义清晰的统一协议
- 先做单窗口、单远程会话场景
- 不在第一版支持多远程目标

### 16.2 跨平台本机采集差异

风险：Windows、macOS、Linux 的命令差异较大。

缓解：

- 平台实现完全隔离
- 统一只在共享层暴露标准化结果

### 16.3 高风险操作误用

风险：结束进程、自定义命令、缓存清理可能带来破坏性影响。

缓解：

- 默认受限
- 明确确认
- 仅白名单
- 允许在设置中关闭相关按钮

## 17. 推荐实施顺序

推荐按以下顺序进入开发：

1. 建立扩展基础骨架与视图注册
2. 定义共享类型和控制器协议
3. 打通本机采集
4. 打通远程 Linux 采集
5. 接入 Webview 仪表盘
6. 接入树视图摘要
7. 接入进程操作与自定义命令
8. 补测试和设置项
